以下、**「A6M-U（CAM720/WEIZHI系）」で現実的に当たりやすい RTSP/ONVIF 情報**を、根拠付きでまとめます（※こちら側では実機に直接アクセスできないので、**“動作報告が複数あるパターン” + “現場で確定させる最短手順”**の形です）。

---

## 1) RTSP ストリームURL（A6M-Uで最有力）

### 推奨（まずこれを試す）

**メイン（高解像度）**

```txt
rtsp://Admin:test12345@192.168.77.23:554/live/ch00_0
```

**サブ（低解像度）**

```txt
rtsp://Admin:test12345@192.168.77.23:554/live/ch00_1
```

根拠：

* iSpyConnect の Jooan DB に **A6M-U が `/live/ch00_1` 系で掲載**されています（同系列で `/live/ch00_0` も広く使われる）。 ([iSpyConnect.com][1])
* Jooan 別モデルで **`/live/ch00_0` が高解像度・`/live/ch00_1` が低解像度**で動作した報告があります。 ([Monocle Forums][2])

> なお OEM/ファーム差で **0/1 が入れ替わる**個体もあり得るので、IS22 側では「両方 probe → 解像度でメイン判定」が堅いです。

### ついでに（候補として残すなら）

あなたが挙げた `/onvif1`, `/11`, `/stream1` は **他OEM系で当たりやすい“別系統”**なので、A6M-U（CAM720系）では優先度低めでOK（ただし保険でprobe対象に入れるのはアリ）。

---

## 2) 認証（ユーザー名・パスワード）

### まず押さえるべき前提

CAM720系は「**ユーザー名をいじれず固定**」なことが多く、Home Assistant コミュニティでは **デフォルトユーザー名が `Admin`、パスワードは CAM720 側で設定したもの**という報告があります。 ([Home Assistant Community][3])

### IS22での実装上のおすすめ

* ユーザー名は **`Admin` と `admin` の両方を試す**（現場差が出る）
* RTSP と Web 認証は **共通になりがち**（上記報告もその前提） ([Home Assistant Community][3])
* 「認証なしRTSP」は期待しない（まず 401/403 を正として扱い、成功時だけ例外扱い）

---

## 3) RTSP 認証方式（Basic/Digest）とトランスポート（TCP/UDP）

### 認証方式

CAM720系の個体によっては Web UI 側に **RTSP認証を Basic に切り替える項目**があり、実際に「Basicにした」との報告があります。 ([Home Assistant Community][3])

**IS22側の現実解**

* まずは通常どおり `rtsp://user:pass@ip:554/...` を投げる
* 401 が返る場合は、クライアント（ffmpeg等）に任せつつ、**Web UIで Basic/Digest 切替があるか確認**（あれば Basic 推奨）

### トランスポート

* **TCP 推奨**（VPN越し/パケットロス環境で安定しやすい）
* UDP は “通れば低遅延” ですが、現場運用は TCP が無難

---

## 4) ONVIF 対応状況（A6M-Uは「ある/ない/半端」が混在しやすい）

あなたの現状は `/onvif/device_service` を **80番で叩いてタイムアウト**なので、次の2点が濃厚です：

### (A) ONVIFが“別ポート”にいる（8899が本命）

低価格カメラ系で **ONVIFが 8899** に来るパターンがあり、Jooan 系でも「**8899にONVIFっぽいものがいる**」という報告があります（ただし実装が不完全で `ActionNotSupported` になることも）。 ([Reddit][4])

**まず試すべきONVIFエンドポイント**

```txt
http://192.168.77.23:8899/onvif/device_service
```

認証はまず：

* user: `Admin`（ダメなら `admin`）
* pass: CAM720で設定した `test12345`

> **注意**：ONVIF “対応” と言っても、PTZや一部APIが死んでる個体があり得ます（`ActionNotSupported` 系）。 ([Reddit][4])

### (B) そもそもONVIF非対応（または無効化）

A6M-Uの購入者レビューで「**ONVIFにも対応してほしい**」という声もあり、**少なくとも一部の流通個体では“ONVIF期待できない”可能性**があります。 ([Amazon Japan][5])

→ IS22としては **ONVIFがダメでも RTSP登録だけで完結できる設計**が安全です。

## すぐ確定させる “最短テスト” コマンド（現場向け）

### RTSP（TCPで確定）

```bash
ffprobe -rtsp_transport tcp -v error -show_streams \
"rtsp://Admin:test12345@192.168.77.23:554/live/ch00_0"
```

```bash
ffprobe -rtsp_transport tcp -v error -show_streams \
"rtsp://Admin:test12345@192.168.77.23:554/live/ch00_1"
```

* `width/height` が大きい方をメイン扱いにすれば、0/1入れ替わり問題を吸収できます。

### ONVIF（8899の生存確認）

```bash
curl -sv --max-time 3 http://192.168.77.23:8899/onvif/device_service
```

* 何らかのHTTP応答が返れば「8899にWebサービスがいる」可能性が上がります（SOAP自体はPOSTが基本）。

## 5) ファームウェア/チップセット（現実的な見方）

* OUI が AltoBeam なのは **Wi-Fi/BLEモジュール側のベンダ**を示していることが多く、映像SoCそのものの特定には直結しないケースが多いです（＝“AltoBeamだからRTSPがこう”とはならない）。
* なお “Jooan A6M” 系で **Thingino（代替FW）に言及される例**はありますが、これは **UART/フラッシュ等の侵襲前提**になりやすいです。 ([GitHub][6])
  → IS22運用で欲しいのはまず **標準FWのRTSP/ONVIF確定**なので、ここは最後の手段扱いでOKです。
